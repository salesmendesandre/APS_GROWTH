# Network Infrastructure Configuration

Before our node can transmit useful data, we need to establish the infrastructure that will listen to and process that data. In this workshop we will use a **Network Server** (such as The Things Network - TTN or a private instance of ChirpStack) to manage our fleet of devices.

## 1. Setting up the Heltec Gateway

The Gateway is the bridge between the electromagnetic spectrum (LoRa) and the IP network (Internet).
- **Internet Connection**: We will configure the Gateway to connect to the Faculty's WiFi network or via Ethernet.
- **Packet Forwarder**: The Gateway's internal software (usually based on the *Semtech UDP Packet Forwarder* or *Basic Station*) must be configured to point to the IP address or domain of our Network Server. The port must be specified (e.g., 1700 for UDP) as well as the Gateway's EUI to identify it on the server.
- **Frequencies**: Ensure that the Gateway is configured to listen on the correct frequency plan (EU868 for Europe).

## 2. Registration on the Network Server

Once the Gateway is routing packets, we must configure the server side:
1. **Gateway Registration**: We register the Gateway in the Network Server console using its EUI. If everything is correct, we will see the status as "Connected".
2. **Application Creation**: We create a logical "Application". This application will group all the environmental nodes that we deploy in the garden.

## 3. Device Registration (OTAA)

To add our ESP32 node to the application, we must register it using the **OTAA (Over-The-Air Activation)** method. The Network Server will generate or ask us for three fundamental keys that we must copy into our node's code:

1. **Device EUI (`DevEUI`)**: A 64-bit globally unique identifier for our ESP32 (similar to a MAC address). Sometimes it is generated by the chip itself, sometimes we assign it manually.
2. **Application EUI / Join EUI (`AppEUI`)**: A 64-bit identifier that identifies which application the device belongs to.
3. **Application Key (`AppKey`)**: A secret AES-128 cryptographic key. It must never be transmitted over the network! It is used to securely negotiate session keys during the *Join Request*.
